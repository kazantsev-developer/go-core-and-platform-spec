## Хэш-таблицы в Go (Maps)

**`map`** — это встроенный ссылочный тип данных, представляющий собой динамическую, высокооптимизированную хэш-таблицу. Она предназначена для хранения пар «ключ-значение» и обеспечивает среднее время поиска, вставки и удаления элементов за константное время $O(1)$.

Стандартная `map` в Go **не является потокобезопасной** (not thread-safe).

---

### Внутреннее устройство (Под капотом)

Переменная типа `map` — это указатель на структуру `hmap` (`runtime/map.go`). На 64-битной архитектуре сама переменная занимает 8 байт.

Основные поля структуры `hmap`:

- **`count`**: количество элементов в мапе прямо сейчас.
- **`flags`**: битовое поле состояния. Используется в том числе для детекции конкурентного доступа (флаг `hashWriting`).
- **`B`**: логарифм количества бакетов (количество бакетов = $2^B$).
- **`buckets`**: указатель на массив бакетов `bmap`. Если мапа пустая, он может быть `nil`.
- **`oldbuckets`**: указатель на старый массив бакетов. Используется только в процессе эвакуации данных (рехэширования).
- **`nevacuate`**: счетчик прогресса эвакуации бакетов.

#### Структура бакета (`bmap`)

Каждый бакет жестко вмещает в себя ровно **8 пар «ключ-значение»**. Компилятор динамически формирует структуру `bmap` во время сборки:

1. Массив `tophash` (8 байт): хранит старшие 8 бит хэша каждого ключа для быстрой предварительной фильтрации при поиске.
2. Массив ключей (8 штук): упакованы подряд.
3. Массив значений (8 штук): упакованы подряд (а не парами ключ-значение, чтобы избежать лишнего оверхеда на выравнивание памяти / padding).
4. Указатель на следующий `overflow bucket` (бакет переполнения): если в бакет хэшируется больше 8 элементов, создается связанный список бакетов.

---

### Алгоритм поиска, роста и рехэширования

#### Как работает поиск ключа

1. Рантайм вычисляет хэш ключа с помощью криптостойкого алгоритма (например, AES-хэширование).
2. Младшие биты хэша определяют индекс бакета в массиве `buckets` (берется остаток от деления хэша на количество бакетов).
3. Старшие 8 бит хэша сравниваются с массивом `tophash` внутри выбранного бакета.
4. Если `tophash` совпал, рантайм проверяет точное равенство ключей через оператор `==`. Если ключи равны — значение возвращается.

#### Критерии роста и эвакуация данных

Мапа начинает увеличивать свой размер при выполнении одного из условий:

1. **Переполнение (Overload):** Среднее количество элементов на один бакет превышает **6.5** (фактор загрузки / _load factor_). Количество бакетов $2^B$ **удваивается**.
2. **Избыток overflow-бакетов:** Элементов мало, но из-за коллизий и частых удалений создалось слишком много бакетов переполнения. Мапа запускает «преобразование на месте» (емкость бакетов не меняется, но данные переупаковываются плотнее).

**Инкрементальный (эвакуация):** Выделяется новый массив бакетов `buckets`, а старый сохраняется в `oldbuckets`. Перенос данных происходит не мгновенно (это вызвало бы фризы рантайма), а **постепенно** — при каждой следующей операции записи или удаления в мапу переносятся ровно 1–2 бакета. Во время эвакуации операции чтения проверяют как новые, так и старые бакеты.

---

### Поведение при операциях (Матрица состояний)

| Состояние мапы                  | Чтение (`v := m[k]`)               | Запись (`m[k] = v`)                           | Удаление (`delete(m, k)`)    | len(m)            |
| :------------------------------ | :--------------------------------- | :-------------------------------------------- | :--------------------------- | :---------------- |
| **Не инициализирована (`nil`)** | Возвращает zero-value              | **Паника** (`assignment to entry in nil map`) | Безопасно (ничего не делает) | `0`               |
| **Инициализирована (`make`)**   | Возвращает значение или zero-value | Успешно записывает                            | Успешно удаляет              | Актуальный размер |

---

### Критические особенности и подводные камни

- **Память мапы не уменьшается:** При вызове `delete()` элементы удаляются, но нижележащая память (массивы бакетов) **не возвращается операционной системе** и не очищается Garbage Collector-ом. Емкость мапы может только расти. Единственный способ физически уменьшить память разросшейся мапы — создать новую мапу через `make` и скопировать туда актуальные данные, дав старой мапе удалиться.
- **Встроенный рантайм-детектор гонок:** В исходном коде функций записи (`mapassign`) выставляется флаг `hashWriting`. Если при чтении (`mapaccess`) или параллельной записи этот флаг уже взведен — рантайм мгновенно роняет приложение через `fatal error: concurrent map read and map write`. Эту ошибку **нельзя перехватить через `recover()`**.
- **Недетерминированный обход (Рандомный Iteration):** Обход мапы через `for k, v := range m` специально возвращает элементы в случайном порядке. При каждом запуске цикла рантайм выбирает случайный стартовый бакет и случайное смещение внутри него. Это сделано для того, чтобы разработчики не закладывались на порядок элементов, так как структура мапы гарантированно меняется при рехэшировании.

### Примеры кода

#### 1. Инициализация мапы с аллокацией (Защита от паники)

```go
package main

import "fmt"

func main() {
	var nilMap map[string]int // nil
	// _ = nilMap["key"]      // Чтение безопасно -> вернет 0

	// ПЛОХО: nilMap["key"] = 10 // ПАНИКА: assignment to entry in nil map

	// ПРАВИЛЬНО: инициализация перед записью
	validMap := make(map[string]int)
	validMap["key"] = 10
	fmt.Println(validMap)
}
```

#### 2. Проверка существования ключа (comma, ok idiom)

```go
package main

import "fmt"

func main() {
	m := map[string]int{"status": 0}

	// Простое чтение не отличает отсутствие ключа от значения 0
	val := m["missing_key"]
	fmt.Println(val) // Вывод: 0

	// ПРАВИЛЬНО: проверка через флаг существования
	val, exists := m["missing_key"]
	if !exists {
		fmt.Println("Ключ реально отсутствует в мапе")
	}
}
```

### Резюме по оптимизации

Стандартная мапа в Go — это структура `hmap`, работающая через массивы бакетов по 8 пар элементов. Чтобы избежать дорогостоящего инкрементального рехэширования (эвакуации данных) при росте хэш-таблицы, в продакшене **всегда нужно инициализировать мапу с заранее известным размером** через `make(map[K]V, hint)`. Память мапы после удаления элементов не уменьшается, а конкурентная запись без мьютекса гарантированно приводит к непредотвращаемому падению всего процесса.
