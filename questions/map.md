## Встроенная проверка на конкурентный доступ в map

**Встроенный механизм детекции конкурентного доступа в `map`** — это рантайм-проверка в исходном коде Go, которая мгновенно завершает работу приложения с фатальной ошибкой (`fatal error: concurrent map read and map write`), если фиксирует одновременную запись и чтение/запись мапы из разных горутин без средств синхронизации.

Этот механизм нужен для **защиты целостности структуры данных в памяти**. В отличие от полноценного динамического детектора гонок (`go test -race`), работающего на базе ThreadSanitizer, проверка в мапе запечена непосредственно в рантайм и активна всегда, даже в продакшн-сборках.

---

### Что хотят услышать интервьюеры (Суть вопроса)

Интервьюер проверяет:

1. Понимает ли кандидат, что стандартная `map` в Go **не является потокобезопасной** (not thread-safe).
2. Различает ли кандидат встроенный рантайм-чек мапы (который паникует через `fatal error`) и полноценный инструмент компилятора `-race`.
3. Знает ли кандидат, как устроена мапа «под капотом» и почему гонка в ней опаснее, чем в слайсе или структуре.

---

### Почему `map` имеет встроенную проверку, а другие типы нет?

#### 1. Сложная внутренняя структура (Угроза повреждения памяти)

Мапа в Go — это сложная хэш-таблица, состоящая из заголовка (`hmap`), массива бакетов (`buckets`), указателей на переполненные бакеты (`overflow buckets`) и флагов состояния.
При перезаписи мапы рантайм может начать процесс **эвакуации данных** (рехэширование/изменение размера мапы). Если в этот момент другая горутина попытается прочитать данные, она улетит по невалидному указателю в случайный участок памяти (Memory Corruption / Segmentation Fault).
_В простых типах (слайс, структура, инт) гонка данных приведет «всего лишь» к неверному значению переменной, но не сломает сам рантайм Go. В мапе же гонка ломает структуру самого языка._

#### 2. Наличие флага состояния (`flags`) в структуре `hmap`

Разработчики Go добавили в скрытую структуру `hmap` специальное битовое поле `flags`. При любой операции записи (`mapassign`) выставляется флаг `hashWriting`. Перед чтением (`mapaccess`) или записью рантайм проверяет этот флаг. Если он уже взведен другой горутиной — рантайм сразу вызывает `throw()`, аварийно завершая процесс. У слайсов или структур такого скрытого системного оверхеда и системных флагов просто нет.

#### 3. Философский дизайн Go: Производительность по умолчанию

Сделать мапу потокобезопасной из коробки (автоматически добавив внутрь мьютекс) означало бы сильно замедлить её работу во всех однопоточных сценариях. Команда Go решила: мапа должна быть быстрой, но если программист ошибается и использует её конкурентно без мьютекса, программа должна не молча выдавать мусор, а сразу жестко падать, сигнализируя об ошибке.

### Примеры поведения кода

#### Ошибочный код (Приведет к падению приложения)

```go
package main

import (
	"fmt"
	"time"
)

func main() {
	m := make(map[string]int)

	// Горутина 1: Пишет в мапу
	go func() {
		for {
			m["key"] = 1 // Выставляет флаг hashWriting
		}
	}()

	// Горутина 2: Читает из мапы
	go func() {
		for {
			_ = m["key"] // Проверяет флаг hashWriting. Если true -> fatal error!
		}
	}()

	time.Sleep(1 * time.Second)
}
// Вывод: fatal error: concurrent map read and map write
```

#### Правильный код (Синхронизация через мьютекс)

```go
package main

import (
	"sync"
)

type SafeMap struct {
	mu sync.RWMutex
	data map[string]int
}

func (s *SafeMap) Set(k string, v int) {
	s.mu.Lock()         // Блокировка на запись
	defer s.mu.Unlock()
	s.data[k] = v
}

func (s *SafeMap) Get(k string) (int, bool) {
	s.mu.RLock()        // Разделяемая блокировка на чтение
	defer s.mu.RUnlock()
	v, ok := s.data[k]
	return v, ok
}
```

### Резюме для собеседования

Встроенный детектор в мапе — это дешевая «подушка безопасности» на уровне рантайма, предотвращающая крах памяти ядра Go. Он не защищает от логических гонок, а лишь заставляет приложение упасть через `fatal error`, которую **нельзя перехватить через `recover()`**. Для безопасной работы в конкурентной среде разработчик обязан использовать `sync.Mutex`, `sync.RWMutex` или специализированную структуру `sync.Map` для read-heavy сценариев.
