## Каналы в Go (Channels)

**Канал** — это встроенный в Go типизированный примитив конкурентности, предназначенный для синхронизации и безопасного обмена данными между горутинами без явного использования блокировок памяти (`sync.Mutex`).

Каналы реализуют модель CSP (Communicating Sequential Processes), где владение данными передается через сообщения.

---

### Внутреннее устройство (Под капотом)

Канал под капотом — это указатель на структуру `hchan` из пакета `runtime/chan.go`. Любая переменная типа `chan` занимает 8 байт (размер указателя).

Основные поля структуры `hchan`:

- **`qcount`**: количество элементов в буфере прямо сейчас.
- **`dataqsiz`**: размерность буфера (емкость, заданная в `make(chan T, cap)`).
- **`buf`**: указатель на кольцевой буфер (массив) для хранения данных (только для буферизованных каналов).
- **`lock`**: внутренний мьютекс (`runtime.mutex`) для защиты полей `hchan` при конкурентных операциях записи/чтения.
- **`recvq` и `sendq`**: связанные списки (очереди) типа `waitq`, содержащие заблокированные горутины (структуры `sudog`), которые ждут чтения или записи в канал.
- **`closed`**: флаг закрытия канала.

---

### Поведение при операциях (Матрица состояний)

| Состояние канала                | Запись (`ch <- x`)                       | Чтение (`<-ch`)                            | Закрытие (`close(ch)`)                    |
| :------------------------------ | :--------------------------------------- | :----------------------------------------- | :---------------------------------------- |
| **Не инициализирован (`nil`)**  | Вечная блокировка                        | Вечная блокировка                          | Паника (`panic: close of nil channel`)    |
| **Открыт, пуст/буфер не полон** | Запись успешна                           | Блокировка (если пуст)                     | Успешно закрывается                       |
| **Открыт, переполнен**          | Блокировка                               | Чтение успешна                             | Успешно закрывается                       |
| **Закрыт**                      | Паника (`panic: send on closed channel`) | Возвращает zero-value и флаг `ok == false` | Паника (`panic: close of closed channel`) |

---

### Типы каналов и механика их работы

#### 1. Небуферизованные каналы (`cap == 0`)

- **Механика:** Точка жесткой синхронизации горутин (rendezvous). Передача данных происходит напрямую из стека пишущей горутины в стек читающей горутины, минуя кольцевой буфер.
- **Поведение:** Отправитель блокируется, пока получатель не придет за данными, и наоборот.

#### 2. Буферизованные каналы (`cap > 0`)

- **Механика:** Асинхронная передача данных через кольцевой массив `buf`.
- **Поведение:** Отправитель не блокируется, пока `qcount < dataqsiz`. При переполнении буфера горутина-отправитель упаковывается в `sudog` и встает в очередь `sendq`. При опустошении буфера горутина-получатель встает в очередь `recvq`.

### Примеры кода

#### 1. Использование comma, ok idiom при чтении

```go
package main

import "fmt"

func main() {
	ch := make(chan int, 1)
	ch <- 42
	close(ch)

	// Правильное чтение из закрытого канала
	val, ok := <-ch
	fmt.Printf("Значение: %d, Открыт: %t\n", val, ok) // Вывод: 42, true

	val, ok = <-ch
	fmt.Printf("Значение: %d, Открыт: %t\n", val, ok) // Вывод: 0, false (канал пуст и закрыт)
}
```

#### 2. Блокировка в одной горутине (Deadlock)

```go
package main

func main() {
	ch := make(chan int, 2)
	ch <- 1
	ch <- 2

	// Ошибка: Запись превышает емкость буфера в одном потоке выполнения
	ch <- 3 // fatal error: all goroutines are asleep - deadlock!
}
```
