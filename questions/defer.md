## Ключевое слово defer в Go

**`defer`** — это встроенное ключевое слово в Go, предназначенное для отложенного вызова функции. Отложенная функция гарантированно выполняется непосредственно перед выходом из окружающей её функции (после выполнения основного тела функции, но до того, как управление вернется вызывающей стороне).

`defer` используется для гарантированного освобождения ресурсов, закрытия дескрипторов, снятия блокировок и обработки паник, независимо от того, завершилась ли функция успешно или с ошибкой (early return, panic).

---

### Ключевые особенности работы defer (Спрашивают на собеседованиях)

1. **Порядок выполнения (LIFO):** Если в одной функции объявлено несколько вызовов `defer`, они складываются в стек и выполняются в **обратном порядке** их объявления (Last In, First Out — «последним пришел, первым ушел»).
2. **Момент вычисления аргументов:** Аргументы отложенной функции вычисляются **в момент объявления** `defer`, а не в момент фактического выполнения самой функции.
3. **Изменение именованных возвращаемых значений:** Если окружающая функция имеет _именованные возвращаемые переменные_ (named return values), отложенная функция имеет к ним доступ и может изменить финальный результат работы функции после выполнения оператора `return`.
4. **Связка с Panic/Recover:** `defer` — это единственное место, где функция `recover()` может перехватить панику (`panic`) и предотвратить падение приложения.

---

### Основные области применения

- **Освобождение системных ресурсов:** Закрытие файлов (`file.Close()`), закрытие сетевых соединений (`resp.Body.Close()`).
- **Конкурентность:** Снятие блокировок мьютексов (`mutex.Unlock()`), сигнализирование в `sync.WaitGroup` (`wg.Done()`).
- **Логирование/Профайлинг:** Фиксация времени выполнения функции (замер duration).
- **Восстановление после сбоев:** Перехват критических ошибок через `recover()`.

---

### Примеры использования

#### 1. Классический паттерн: Управление ресурсами и стек (LIFO)

```go
package main

import (
	"fmt"
	"os"
)

func main() {
	file, err := os.Open("example.txt")
	if err != nil {
		fmt.Printf("ошибка открытия файла: %v\n", err)
		return
	}
	// Использование defer гарантирует закрытие дескриптора при любом выходе из main
	defer file.Close()

	// Демонстрация LIFO: этот дефер выполнится ПЕРЕД file.Close()
	defer fmt.Println("1. Первая отложенная строка (LIFO - выполнится раньше)")
	defer fmt.Println("2. Вторая отложенная строка (LIFO - выполнится позже)")

	fmt.Println("Выполнение основной логики функции...")
}
```

#### 2. Вычисление аргументов vs Именованный return (Продвинутый уровень)

```go
package main

import "fmt"

func main() {
	fmt.Printf("Результат функции с именованным return: %d\n", namedReturnDemo())
	argumentEvaluationDemo()
}

// Пример 1: Изменение возвращаемого значения
func namedReturnDemo() (result int) {
	result = 10
	defer func() {
		result += 5 // Напрямую изменяет именованную переменную перед выходом из функции
	}()
	return result // Функция вернет 15, а не 10
}

// Пример 2: Момент вычисления аргументов
func argumentEvaluationDemo() {
	x := 10
	// Аргумент `x` копируется СЕЙЧАС (значение 10)
	defer fmt.Printf("Значение x в defer: %d\n", x)

	x = 20
	fmt.Printf("Значение x в конце функции: %d\n", x)
	// На выходе из функции defer напечатает 10, а не 20!
}
```

#### 3. Восстановление после паники (Panic & Recover)

```go
package main

import "fmt"

func process() {
	// recover() работает только внутри defer
	defer func() {
		if r := recover(); r != nil {
			fmt.Printf("Успешно перехвачена паника: %v. Выполняем cleanup...\n", r)
		}
	}()

	fmt.Println("Старт обработки данных...")
	panic("критическая ошибка в рантайме") // Имитируем панику
	fmt.Println("Этот код никогда не выполнится")
}

func main() {
	process()
	fmt.Println("Программа продолжает работать благодаря recover в defer!")
}
```

---

### Резюме для собеседования

Использование `defer` делает код безопасным и читаемым, предотвращая утечки ресурсов. Однако стоит помнить, что в высоконагруженных циклах (например, `for`) использовать `defer` напрямую внутри цикла нельзя, так как стек деферов будет расти до тех пор, пока вся окружающая функция не завершится, что может привести к исчерпанию памяти. Для этого логику итерации выносят в отдельную функцию (хелпер).
