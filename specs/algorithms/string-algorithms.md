## Спецификация строковых алгоритмов: Низкоуровневая структура string, оптимизация поиска подстрок и управление аллокациями памяти в Go

Обработка строковых данных в высоконагруженных бэкенд-сервисах является одной из самых ресурсоемких операций. Строки в Go неизменяемы (immutable), что при неоптимальном кодировании приводит к массовому выделению временных объектов в куче (Heap Allocation) и перегрузке Сборщика мусора. Понимание физического представления строк в памяти хоста и применение эффективных алгоритмов поиска подстрок позволяют снизить временную сложность до линейной O(N) при сохранении жестких рамок Zero-Allocation.

---

### 1. Физическая структура string под капотом рантайма Go

На уровне рантайма Go строка не является массивом символов или аналогом класса из других языков. Физически это легковесная 16-байтная структура `stringStruct`, объявленная в системном пакете `runtime/string.go`.

#### Анатомия структуры памяти

Заголовок строки на 64-битных архитектурах занимает ровно 16 байт и состоит из двух полей:

- `str unsafe.Pointer` — 8-байтный прямой указатель на первый байт базового массива байт (`[ ]byte`), расположенного в памяти.
- `len int` — 8-байтное целое число, фиксирующее точную длину строки в байтах (а не в символах).

#### Ограничения неизменяемости (Immutability)

Поскольку базовый массив байт строки защищен от модификаций, любая наивная операция конкатенации вида `str += "text"` внутри цикла вынуждает рантайм вызывать функцию `runtime.concatstrings`. Она выделяет совершенно новый блок памяти в куче, копирует туда старые байты и дописывает новые. Это порождает квадратичную сложность по памяти и времени.

Для обхода этого ограничения в промышленной разработке применяется тип `strings.Builder`. Под капотом он использует стандартный слайс байт, расширяемый по формуле роста слайсов Go, минимизируя обращения к системному аллокатору за счет предварительного резервирования памяти через метод `Grow()`.

### 2. Алгоритм поиска подстроки Рабина-Карпа (Rabin-Karp)

Алгоритм Рабина-Карпа решает задачу поиска подстроки (pattern) длины M внутри строки (text) длины N с использованием математического механизма хэширования.

#### Концепция Скользящего хэша (Rolling Hash)

Вместо последовательного посимвольного сравнения на каждом шаге (что дает худшую сложность O(N \* M)), алгоритм один раз вычисляет числовой хэш от искомого паттерна и последовательно сравнивает его с хэшем текущего окна в строке.

- Вычисление хэша для нового сдвинутого окна происходит за константное время O(1) с помощью полиномиального хэширования. Рантайм не пересчитывает всю строку заново: он просто вычитает математический вклад первого выбывающего символа слева и прибавляет вклад нового входящего символа справа.
- Если числовые хэши совпали, алгоритм выполняет посимвольную проверку (для защиты от коллизий хэш-функции).

Этот алгоритм является базовым и нативно используется в стандартной библиотеке Go в функции `strings.Index()`.

#### Промышленная реализация алгоритма Рабина-Карпа на Go

```go
package main

const primeBase = 101 // Базовое простое число для полиномиального хэша

// RabinKarp реализует поиск подстроки за амортизированное время O(N + M) и O(1) по памяти
func RabinKarp(text, pattern string) int {
	n := len(text)
	m := len(pattern)

	if m == 0 || n < m {
		return -1
	}

	var patternHash, currentHash uint32
	var highestPower uint32 = 1

	// Расчет веса старшего разряда для сдвига окна
	for i := 0; i < m-1; i++ {
		highestPower = (highestPower * primeBase)
	}

	// Предварительный расчет хэша паттерна и первого окна текста
	for i := 0; i < m; i++ {
		patternHash = patternHash*primeBase + uint32(pattern[i])
		currentHash = currentHash*primeBase + uint32(text[i])
	}

	// Скольжение окна по тексту
	for i := 0; i <= n-m; i++ {
		if patternHash == currentHash {
			// Верификация при совпадении хэшей для защиты от коллизий
			if text[i:i+m] == pattern {
				return i // Подстрока локализована
			}
		}

		// Вычисление хэша для следующего окна за O(1)
		if i < n-m {
			currentHash = (currentHash - uint32(text[i])*highestPower)*primeBase + uint32(text[i+m])
		}
	}

	return -1 // Подстрока отсутствует
}
```

### 3. Алгоритм Кнута-Морриса-Пратта (KMP) и префикс-функция

Алгоритм Кнута-Морриса-Пратта (Knuth-Morris-Pratt) осуществляет поиск подстроки за детерминированное линейное время O(N + M) в худшем сценарии. В отличие от наивного подхода, KMP полностью исключает холостые откаты назад по исходному тексту при обнаружении несовпадения символов.

#### Механика префикс-функции (Pi-массив)

Перед стартом поиска алгоритм компилирует искомый паттерн в массив чисел — префикс-функцию (длиной M). Каждое значение `pi[i]` фиксирует длину максимального собственного префикса суффикса для подстроки `pattern[0:i]`.

Если во время сканирования основного текста на позиции J происходит сбой совпадения с паттерном, алгоритм считывает значение из `pi[j-1]` и мгновенно сдвигает индекс паттерна на эту позицию. KMP знает, какая часть строки уже гарантированно совпала, и продолжает движение только вперед.

#### Реализация алгоритма KMP на Go

```go
package main

// Вычисление префикс-функции — временная сложность O(M), пространственная O(M)
func computePrefixFunction(pattern string) []int {
	m := len(pattern)
	pi := make([]int, m)
	k := 0

	for i := 1; i < m; i++ {
		for k > 0 && pattern[k] != pattern[i] {
			k = pi[k-1]
		}
		if pattern[k] == pattern[i] {
			k++
		}
		pi[i] = k
	}
	return pi
}

// KMPSearch выполняет поиск подстроки за O(N + M) по времени
func KMPSearch(text, pattern string) int {
	n := len(text)
	m := len(pattern)

	if m == 0 {
		return 0
	}
	if n < m {
		return -1
	}

	pi := computePrefixFunction(pattern)
	q := 0

	for i := 0; i < n; i++ {
		for q > 0 && pattern[q] != text[i] {
			q = pi[q-1]
		}
		if pattern[q] == text[i] {
			q++
		}
		if q == m {
			return i - m + 1 // Подстрока найдена, возвращаем стартовый индекс
		}
	}
	return -1
}
```

### 4. Обработка кодировки UTF-8 и оверхед типа rune в Go

Применение алгоритмов Рабина-Карпа или KMP напрямую к байтовым индексам строк (`text[i]`) валидно только для символов таблицы ASCII, где один символ строго равен 1 байту.

#### Проблема многобайтовых кодировок

Кириллица, эмодзи и спецсимволы в формате UTF-8 занимают от 2 до 4 байт. Наивный побайтовый обход разрушит логическую структуру символа. Для корректной обработки многобайтового текста в Go используется тип **`rune`** (синоним `int32`), представляющий собой кодовую точку Юникода (Code Point).

#### Скрытые аллокации при приведении типов

Приведение строки к слайсу рун `runes := []rune(text)` для последующего поиска — это тяжелая операция, которая заставляет рантайм полностью декодировать всю строку, выделить новый массив в куче и скопировать туда данные. Это уничтожает концепцию Zero-Allocation.

Для сохранения высокой производительности в Highload-системах обход строк по кодовым точкам Юникода должен выполняться итеративно через цикл `range` или системные функции пакета `unicode/utf8` (например, `utf8.DecodeRuneInString`), которые работают по указателям без выделения памяти в куче.

### 5. Методология микробенчмаркинга строковых алгоритмов

Контроль за отсутствием аллокаций в куче при работе со строковыми методами инициализируется стандартным микробенчмарком:

```go
package main

import "testing"

func BenchmarkKMPSearch(b *testing.B) {
	text := "преобразование строк в высоконагруженных бэкенд сервисах на языке go требует оптимизации"
	pattern := "сервисах"

	b.ResetTimer()
	for i := 0; i < b.N; i++ {
		_ = KMPSearch(text, pattern)
	}
}
```

Команда выполнения тестов производительности в CLI:

```bash
go test -bench=BenchmarkKMPSearch -benchmem
```

Критерий промышленной оценки: показатели `B/op` и `allocs/op` должны быть минимальными (в идеале равными `0`), подтверждая отсутствие утечек контекста функций в кучу хоста и исключая деградацию рантайма из-за фаз Stop-The-World фонового Garbage Collector.
