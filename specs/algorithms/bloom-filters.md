## Спецификация вероятностных структур данных: Проектирование и оптимизация Фильтров Блума на Go

В высоконагруженных распределенных системах (Highload) проверка наличия элемента в гигантских массивах данных (миллионы ID пользователей, URL-адресов или транзакций) стандартными методами через хэш-таблицы порождает избыточный оверхед по памяти. Фильтр Блума (Bloom Filter) — это пространственно эффективная вероятностная структура данных, предназначенная для мгновенной проверки принадлежности элемента к множеству. Фильтр Блума позволяет определить, что элемент либо гарантированно отсутствует в структуре данных, либо с определенной вероятностью в ней присутствует.

---

### 1. Архитектурная концепция и математический базис

Фильтр Блума полностью отказывается от физического хранения самих элементов. Вместо этого в памяти выделяется плоский битовый массив фиксированного размера.

#### Принцип работы и ложноположительные срабатывания (False Positives)

- **Добавление элемента:** Элемент (например, строка) пропускается через K независимых хэш-функций. Каждая хэш-функция генерирует числовой индекс в рамках размера битового массива. Биты по этим индексам принудительно устанавливаются в 1.
- **Проверка элемента:** Строка снова хэшируется теми же K функциями. Если хотя бы один бит по полученным индексам равен 0, алгоритм выдает математически гарантированный ответ: **элемента точно нет в базе**. Если все биты равны 1, система выдает ответ: **элемент с высокой вероятностью есть в базе**.
- **False Positive:** Ситуация, когда из-за пересечения хэшей от разных строк одни и те же биты были установлены в 1. Алгоритм может сказать, что элемент присутствует, хотя его никогда не добавляли. Вероятность такой ошибки настраивается инженером при проектировании за счет изменения размера битового массива и количества хэш-функций. Ложноотрицательных срабатываний (False Negatives) в фильтре Блума не существует в принципе.

#### Инфраструктурное применение в СУБД (Защита дискового ввода-вывода)

В промышленных базах данных (LSM-Tree движки в RocksDB, Cassandra или СУБД ScyllaDB) Фильтр Блума держится целиком в оперативной памяти (RAM) перед дисковыми файлами данных (SSTables) [INDEX].

Если пользователь запрашивает отсутствующий ключ, Фильтр Блума мгновенно определяет его отсутствие в памяти за O(1) времени. Это полностью избавляет систему от холостых, тяжелых и медленных дисковых чтений (Random I/O) [INDEX], сохраняя высокую пропускную способность СУБД.

### 2. Реализация Фильтра Блума на Go на базе пакета math/big

Для эффективного управления битами в памяти Go без накладных расходов на структуры данных применяется тип `big.Int` из стандартного пакета `math/big`, который под капотом оперирует массивом машинных слов и побитовыми операциями.

```go
package main

import (
	"hash/fnv"
	"math/big"
)

type BloomFilter struct {
	bitSet   *big.Int // Эффективное хранилище битов в памяти
	m        uint     // Фактический размер битового массива в битах
	k        uint     // Количество независимых хэш-функций
}

// NewBloomFilter инициализирует структуру под заданные параметры
func NewBloomFilter(m, k uint) *BloomFilter {
	return &BloomFilter{
		bitSet: new(big.Int),
		m:      m,
		k:      k,
	}
}

// Базовая полиномиальная хэш-функция для генерации дополнительных индексов
func (bf *BloomFilter) getHash(data []byte, seed uint) uint {
	h := fnv.New32a()
	h.Write(data)
	hash := h.Sum32()
	// Внесение соли (seed) для имитации независимости K хэш-функций
	return uint(hash+uint32(seed)) % bf.m
}
```

### 3. Высокопроизводительные операции Add и Contains

Реализация методов расширения фильтра Блума на Go базируется на побитовых сдвигах и логическом побитовом И / ИЛИ над структурой `big.Int`. Это исключает накладные расходы на итерации по массивам и выполняется за минимальное количество тактов CPU.

```go
package main

// Add вносит элемент в фильтр Блума, устанавливая K битов в состояние 1
func (bf *BloomFilter) Add(data []byte) {
	for i := uint(0); i < bf.k; i++ {
		idx := bf.getHash(data, i)
		// Побитовый сдвиг 1 на индекс idx и принудительное ИЛИ (|) с текущим битовым набором
		bf.bitSet.Or(bf.bitSet, new(big.Int).Lsh(big.NewInt(1), idx))
	}
}

// Contains проверяет присутствие элемента.
// Если хотя бы один бит равен 0, элемент гарантированно отсутствует (возвращает false).
func (bf *BloomFilter) Contains(data []byte) bool {
	for i := uint(0); i < bf.k; i++ {
		idx := bf.getHash(data, i)
		// Проверка состояния конкретного бита через операцию побитового И (&)
		temp := new(big.Int).Lsh(big.NewInt(1), idx)
		andResult := new(big.Int).And(bf.bitSet, temp)

		if andResult.Cmp(temp) != 0 {
			return false // Математически подтверждено отсутствие элемента
		}
	}
	return true // Элемент с высокой степенью вероятности присутствует
}
```

### 4. Оптимизация хэширования: Схема Кирша-Митценмахера

В реальном промышленном Highload-бэкенде вычисление $K$ независимых хэш-функций (например, запуск 5–8 отдельных криптографических хэшей типа SHA-256 или тяжелых функций вроде MurmurHash3) порождает колоссальный оверхед на CPU.

Для оптимизации применяется математическая схема Кирша-Митценмахера (Kirsch-Mitzenmacher). Она доказывает, что для генерации $K$ уникальных индексов достаточно вычислить всего две хэш-функции ($H_1(x)$ и $H_2(x)$), а все последующие индексы рассчитывать линейной комбинацией по формуле:

$$g_i(x) = H_1(x) + i \times H_2(x) \pmod m$$

Это позволяет свести накладные расходы на хэширование к константному времени вычисления всего двух базовых значений, полностью разгружая процессор и сохраняя математическую точность вероятностной структуры.

### 5. Методология микробенчмаркинга вероятностных структур

Контроль за эффективностью утилизации памяти и отсутствием скрытых утечек в кучу при побитовых операциях `big.Int` осуществляется через написание бенчмарков:

```go
package main

import (
	"strconv"
	"testing"
)

func BenchmarkBloomFilterAddAndContains(b *testing.B) {
	// Инициализация фильтра Блума: 100 000 битов, 5 хэш-функций
	bf := NewBloomFilter(100000, 5)

	// Предварительное наполнение структуры
	for i := 0; i < 1000; i++ {
		bf.Add([]byte("user_id_" + strconv.Itoa(i)))
	}

	b.ResetTimer()
	for i := 0; i < b.N; i++ {
		// Тестирование операции проверки наличия элемента в памяти
		_ = bf.Contains([]byte("user_id_500"))
	}
}
```

Команда запуска бенчмарка в CLI:

```bash
go test -bench=BenchmarkBloomFilterAddAndContains -benchmem
```

Критерий прохождения контроля:

- Скорость выполнения операции `Contains` должна находиться в пределах наносекундного диапазона.
- Объем аллокаций в куче (`allocs/op`) должен быть минимальным, доказывая, что основные побитовые маски выполняются в рамках регистров и быстрого стека горутины.
