## Архитектура процессов CI/CD и декларативного деплоймента

Конвейер непрерывной интеграции и доставки (CI/CD) предназначен для автоматизации процессов сборки, тестирования и развертывания программного обеспечения. При переходе от императивного управления standalone-серверами к декларативной оркестрации коренным образом меняется логика взаимодействия агентов сборки с целевой инфраструктурой исполнения.

---

### Разделение зон ответственности: Назначение конфигурационных файлов

В отказоустойчивой промышленной архитектуре зоны ответственности между инструментами автоматизации сборки и средами выполнения строго разграничены:

- **`.gitlab-ci.yml` (Конвейер автоматизации):** Является статической инструкцией для централизованного оркестратора сборки. Файл описывает последовательность стадий (stages), правила запуска линтеров, сборку исполняемых бинарных файлов, упаковку слоев Docker-образов и их отправку в удаленный реестр (Docker Registry). Данный манифест не управляет топологией сети на серверах и не хранит конфигурацию постоянных томов хоста.
- **`docker-compose.yml` (Локальный оркестратор процессов):** Представляет собой декларативное описание стека взаимосвязанных контейнеров, запускаемых на конкретном Docker-демоне хоста. Определяет сопоставление портов, точки монтирования файловой системы (Volumes) и локальные переменные окружения. Инструмент автономен и ничего не знает об этапах тестирования кода внутри систем контроля версий.

---

### Сравнительный анализ подходов к деплою приложений

При императивном подходе на базе Docker-Compose управление процессами строится на прямых командах интерфейса командной строки, таких как `docker-compose up -d`. Взаимодействие требует предоставления терминального SSH-доступа на целевой хост с максимальными привилегиями пользователя `root`. Логическая изоляция различных контуров сред при такой схеме достигается исключительно физическим разделением проектов по разным виртуальным машинам. Управление базами данных часто происходит внутри единого сетевого стека совместно с кодом бизнес-логики.

Декларативный подход в рамках Kubernetes Workflow полностью меняет эту схему. Развертывание приложений осуществляется путем отправки структурированного манифеста в формате YAML на центральную точку доступа API. Взаимодействие агента автоматизации с кластером строго ограничивается ролевыми токенами доступа (`ServiceAccount`), что исключает компрометацию хостовой операционной системы. Изоляция сред разработки и продакшна переводится на уровень виртуальных пространств имен (`Namespaces`) внутри единого пула железа. Архитектура требует полного выноса баз данных за контур бесстейт-кода на выделенные persistent-мощности.

### Деградация инфраструктуры при императивном развертывании

Использование Docker-Compose в качестве основного инструмента развертывания на продакшн-серверах порождает ряд низкоуровневых проблем на уровне операционной системы, дисковой подсистемы и безопасности:

#### 1. Компрометация доступов

Императивный деплой через GitLab Runner требует передачи SSH-ключей с правами суперпользователя (`root`) в переменные окружения сборщика. Компрометация учетных данных на стороне CI/CD предоставляет злоумышленнику неограниченный контроль над ядром операционной системы целевого хоста.

#### 2. Блокировка сетевых интерфейсов и зомби-процессы

Команда `docker-compose down`, вызываемая для обновления версии кода, отправляет процессам внутри контейнера сигнал `SIGTERM`, а по истечении таймаута — `SIGKILL`. Если рантайм приложения некорректно обрабатывает сигналы завершения, процессы переходят в состояние зомби (`zombie state`). Сетевой сокет операционной системы хоста не высвобождается вовремя, порождая ошибку удержания порта (зомби-порт, когда `docker-proxy` продолжает находиться в статусе `LISTEN` на интерфейсе при фактически мертвом контейнере). Это требует ручной перезагрузки всего Docker-демона через `systemctl restart docker`.

#### 3. Неконтролируемая фрагментация ресурсов хоста

Базы данных (MongoDB, PostgreSQL) чувствительны к операциям ввода-вывода (IOPS) и потреблению оперативной памяти. Запуск СУБД в одном Compose-файле с бизнес-логикой создает риски ресурсного голодания. При утечке памяти в коде приложения механизм `OOM Killer` операционной системы начинает аварийно завершать процессы на хосте, подставляя под удар контейнер базы данных, что ведет к повреждению таблиц и индексов.

Кроме того, частые перезапуски стека оставляют на накопителе несвязанные анонимные тома (`orphan volumes`) и зависшие слои образов (`dangling images`), вызывая переполнение диска (`No space left on device`).

### Декларативная изоляция и безопасность в Kubernetes Workflow

При переходе на Kubernetes конвейер CI/CD полностью изолируется от операционной системы серверов. GitLab Runner взаимодействует исключительно с центральной точкой входа кластера — `kube-apiserver`.

#### 1. Ролевой доступ (RBAC)

Авторизация агента сборки реализуется через механизм управления доступом на основе ролей. Пайплайн использует ограниченный токен `ServiceAccount`, наделенный правами на модификацию ресурсов только в строго определенном `Namespace` (например, `dev` или `prod`). Доступ к системным вызовам ядра хоста или соседним изолированным средам для сборщика физически заблокирован.

#### 2. Автономное выполнение RollingUpdate

После отправки декларативной команды (например, `kubectl apply -f deployment.yaml`) API-сервер фиксирует изменения в транзакционном хранилище и освобождает ресурсы GitLab Runner. Внутренний контроллер кластера берет управление на себя: изолированно скачивает образ, запускает новые Поды на свободных участках памяти, тестирует их готовность через `Readiness Probe` и только после этого плавно гасит старые контейнеры, исключая появление Downtime.

#### 3. Вынос СУБД за контур бесстейт-кода

В промышленной архитектуре базы данных полностью исключаются из эфемерных контейнеров приложений. СУБД разворачивается отдельно на выделенных мощностях или persistent-хранилищах хоста с настроенной репликацией. Контейнеры приложений подключаются к ней удаленно через защищенную строку соединения (Connection String), передаваемую в переменные окружения (`ENV`). Обновление или падение контейнеров с кодом приложения никак не затрагивает слой хранения данных.

### Низкоуровневая архитектура GitLab Runner

**GitLab Runner** — это изолированная системная служба, предназначенная для обработки и выполнения задач (jobs), описанных в конвейере CI/CD. Сам по себе раннер является бесстейт-процессом, который опрашивает API GitLab на наличие новых задач, скачивает код репозитория и передает команды на исполнение во внутреннюю среду.

#### Исполнители задач (Executors)

Тип исполнителя определяет изолированность среды, в которой будут выполняться команды скрипта:

- **Shell Executor:** Запускает команды непосредственно внутри терминала той операционной системы, где физически установлена служба раннера. Все задачи делят между собой общие бинарные файлы хоста, системные зависимости и права доступа. Данный тип используется для выполнения сложных инфраструктурных сценариев деплоя, требующих прямых вызовов локальных утилит хоста или SSH-соединений с удаленными узлами. Главный недостаток — риск загрязнения окружения (`environment pollution`) оставшимися после сборки артефактами.
- **Docker Executor:** Создает абсолютно чистый, изолированный контейнер (например, на базе минимального образа `alpine:latest`) под каждую конкретную задачу пайплайна и полностью уничтожает его вместе со всеми временными файлами сразу после завершения шага. Это гарантирует неизменяемость сред сборки, чистоту зависимостей и исключает влияние задач друг на друга. Внутри такого контейнера проект компилируется изолированно, используя строго заданный в манифесте образ рантайма (например, `golang:1.23`).

### Алгоритм регистрации нового Docker-раннера

Для интеграции нового проекта в централизованную систему автоматизации процедура привязки изолированного Docker-исполнителя разделяется на два этапа:

#### Шаг 1. Настройка токена аутентификации в веб-интерфейсе

Сначала необходимо перейти в веб-интерфейс GitLab по пути `Project`, затем открыть `Settings` и зайти в раздел `CI/CD`. В появившейся секции `Runners` инициализируется создание нового триггера через команду `New project runner`. В параметрах конфигурации в обязательном порядке указывается тег `docker` — это жесткий идентификатор, по которому планировщик пайплайнов определяет, какому типу исполнителя делегировать задачу. После сохранения система генерирует уникальный токен авторизации (строка с префиксом `glrt-`), который доступен для копирования один раз.

#### Шаг 2. Интерактивная регистрация службы на сервере сборщика

Подключение к целевому серверу сборщика выполняется по защищенному каналу, после чего вызывается утилита регистрации с прямой передачей базовых флагов:

```bash
gitlab-runner register \
  --url http://<gitlab-instance-url> \
  --token <auth-token-glrt> \
  --executor docker \
  --description "production-application-runner"
```

После вызова бинарного файла запускается внутренний диалог рантайма, требующий подтверждения системных параметров:

1. `Enter the GitLab instance URL`: Пропускается нажатием кнопки `Enter`, так как адрес хоста корректно передан через флаг `--url`.
2. `Enter a name for the runner`: Пропускается нажатием кнопки `Enter`, локальный конфиг сохраняет значение из флага `--description`.
3. `Enter tags for the runner`: Пропускается нажатием кнопки `Enter`, так как теги жестко зафиксированы в базе данных GitLab на этапе генерации токена.
4. `Enter an executor`: Пропускается нажатием кнопки `Enter`, подтверждая выбор контейнеризированного исполнителя.
5. `Enter the default Docker image`: Вводится значение `alpine:latest`. Это базовый легковесный слой (~5 МБ), который рантайм раннера будет использовать по умолчанию для развертывания стартового контейнера и выполнения базовых служебных команд пайплайна (таких как `echo`, `ls` или проверка переменных окружения) перед тем, как запустить кастомный контейнер сборки.

### Системные команды верификации состояния агентов сборки

Контроль за работоспособностью запущенных демонов автоматизации осуществляется через CLI-интерфейс службы `gitlab-runner` на сервере.

Проверка текущего статуса системного демона в операционной системе:

```bash
gitlab-runner status
```

Вывод полного списка зарегистрированных локальных исполнителей и проверка их токенов:

```bash
gitlab-runner list
```

Принудительная перезагрузка демона для применения изменений в конфигурационном файле `/etc/gitlab-runner/config.toml`:

```bash
gitlab-runner restart
```
