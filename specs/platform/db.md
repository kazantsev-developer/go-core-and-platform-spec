## Внутреннее устройство и спецификация движков баз данных (Database Internals)

Системы управления базами данных (СУБД) классифицируются по архитектуре хранения, моделям данных и механизмам обеспечения консенсуса. Выбор СУБД для промышленной эксплуатации определяется требованиями к транзакционности, характеру нагрузки (OLTP/OLAP) и топологии распределенной системы.

---

### 1. Архитектура слоев хранения на диске: B-Tree против LSM-Tree

Низкоуровневая производительность СУБД на операциях записи и чтения зависит от структуры данных, используемой движком хранения (Storage Engine) для организации файлов на диске.

#### B-Tree / B+Tree (Реляционные СУБД: PostgreSQL, MySQL, SQLite)

Движки на базе B+Tree организуют данные на диске в виде сбалансированного дерева, разбитого на страницы фиксированного размера (обычно 8 КБ в PostgreSQL). В B+Tree значения (строки таблицы) хранятся исключительно в листовых узлах (leaves), а внутренние узлы содержат только ключи и указатели для маршрутизации поиска.

- **Механика записи (In-place updates):** При модификации или вставке строки движок находит целевую страницу на диске, считывает ее в буферный пул памяти (Buffer Pool), изменяет байты и помечает страницу как «грязную» (dirty page). Затем фоновый процесс (Checkpointer) асинхронно сбрасывает (flushes) грязные страницы обратно на диск, перезаписывая их поверх старых данных.
- **Проблема произвольного ввода-вывода (Random I/O):** Поскольку новые строки могут относиться к разным страницам дерева, запись превращается в хаотичные прыжки по диску (Random I/O). Это создает оверхед на дисковой подсистеме. Для предотвращения повреждения данных при сбое питания до перезаписи страницы движок обязан атомарно зафиксировать операцию в WAL (Write-Ahead Log) файле последовательным методом (Sequential I/O).
- **Специфика чтения:** Поиск по ключу выполняется за константное время $O(\log N)$ и требует минимального количества чтений с диска, что делает B+Tree идеальным решением для транзакционных нагрузок с высокой интенсивностью точечного чтения (**OLTP**).

#### LSM-Tree (Log-Structured Merge-Tree) (NoSQL СУБД: Cassandra, ScyllaDB, ClickHouse)

LSM-Tree полностью отказывается от модификации данных на диске «на месте» (In-place updates) в пользу последовательной дозаписи (Append-only).

- **Механика записи:** Входящая запись атомарно пишется в WAL на диске (для персистентности) и одновременно вставляется в упорядоченную структуру данных в оперативной памяти — **MemTable** (обычно реализуется через Skip-List). При достижении лимита размера (например, 64 МБ) MemTable преобразуется в неизменяемый файл на диске — **SSTable (Sorted String Table)**. Данные в SSTable строго отсортированы по ключу. Запись всегда происходит последовательно (Sequential I/O), что максимизирует пропускную способность дисковой подсистемы.
- **Проблема фрагментации и Уплотнение (Compaction):** Поскольку обновления и удаления — это просто новые записи в новые SSTable (удаление помечается маркером _Tombstone_), на диске скапливается множество дубликатов одной строки. Для очистки запускается фоновый процесс **Compaction**: он считывает несколько SSTable, сливает их (Merge Sort), оставляет только актуальные версии ключей и удаляет Tombstone-маркеры, записывая новую консолидированную SSTable.
- **Специфика чтения (Read Amplification):** При чтении ключа движок вынужден проверять MemTable, а затем последовательно сканировать SSTable от свежих к старым. Для оптимизации применяются **Фильтры Блума (Bloom Filters)** — вероятностные структуры данных в памяти, которые позволяют мгновенно узнать, что ключа точно _нет_ в конкретной SSTable, предотвращая холостые обращения к диску. LSM-Tree оптимизирован для сверхвысоких нагрузок на запись (**Write-Heavy**).

### 2. Сравнительный анализ реляционных и нереляционных моделей

#### Реляционная модель (PostgreSQL)

Данные жестко структурированы в виде связанных таблиц с фиксированной схемой (Schema-on-write). Связи валидируются на уровне движка через внешние ключи (Foreign Keys).

- **Многоверсионность (MVCC):** Для обеспечения конкурентного доступа PostgreSQL использует MVCC (Multi-Version Concurrency Control). При обновлении строки физически не изменяются. Вместо этого создается новая версия строки. Каждая строка содержит скрытые системные атрибуты `xmin` (ID транзакции, создавшей строку) и `xmax` (ID транзакции, удалившей/обновившей строку). Читающие транзакции видят только те версии строк, которые были зафиксированы на момент их старта. Старые, ненужные версии строк вычищаются фоновым процессом **VACUUM**. Если VACUUM не справляется, возникает транзакционное раздувание таблиц (_Table Bloat_), снижающее скорость сканирования диска.

#### Документо-ориентированная модель (MongoDB)

Данные хранятся в виде гибких документов (BSON — бинарный JSON). Схема валидируется на уровне приложения (Schema-on-read). Connections/Joins на уровне движка отсутствуют или являются ресурсоемкими операциями (`$lookup`). Данные денормализуются: связанные сущности вкладываются внутрь одного документа (Embedded Documents), что позволяет извлечь весь контекст объекта за один атомарный запрос к диску без операций сборки из разных таблиц. Применяется в распределенных системах с горизонтальным масштабированием (Шардирование).

### 3. Индексы: Физическая структура и семантика работы

Индексы — это дополнительные структуры данных, создаваемые для оптимизации поиска. Они ускоряют чтение, но замедляют операции записи, так как при каждом `INSERT/UPDATE` движок обязан атомарно модифицировать дерево индекса.

- **B-Tree Индексы:** Классическая структура для поиска по точному совпадению или диапазону значений. Движок проходит от корня к листу дерева индекса, находит ключ и извлекает физический указатель на строку в файле таблицы (в PostgreSQL это `TID` — Tuple ID, состоящий из номера страницы и смещения внутри нее).
- **Покрывающие индексы (Covering Indexes / Index-Only Scan):** Если в секции `SELECT` запрашиваются только те поля, которые _уже входят_ в состав составного индекса, движок полностью отказывается от обращения к файлу таблицы на диске. Данные извлекаются напрямую из структуры дерева индекса (Index-Only Scan), что сокращает дисковый ввод-вывод в разы.
- **Сканирование: Index Scan против Sequential Scan (Seq Scan):** Наличие индекса не гарантирует его использование. Если планировщик запросов (Query Planner) на основе собранной статистики видит, что запрос возвращает значительную часть таблицы (например, более 20-30%), он принудительно переключается на полный проход по таблице — **Sequential Scan**. Чтение подряд всех страниц диска (Sequential I/O) в данном сценарии выполняется быстрее, чем хаотичные прыжки по индексу за каждой отдельной строкой (Random I/O).

### 4. Распределенные системы: CAP и PACELC Теоремы

При выходе СУБД за пределы одного сервера в распределенную среду (кластеры), в силу вступают фундаментальные физические ограничения сетевых трансляций.

#### CAP-Теорема

В любой распределенной системе при возникновении сетевого расщепления (Network Partition) возможно обеспечить только два свойства из трех:

- **Consistency (Согласованность):** Любое чтение с любого узла возвращает самую свежую запись либо ошибку.
- **Availability (Доступность):** Любой незавышенный узел всегда возвращает успешный ответ (без гарантии свежести данных).
- **Partition Tolerance (Устойчивость к разделению):** Система продолжает функционировать, несмотря на потерю связи между узлами. Сетевые сбои в реальности неизбежны, поэтому любая распределенная СУБД обязана обладать свойством **P**, выбирая строгий баланс: либо **CP**, либо **AP**.

#### PACELC-Теорема (Расширение CAP)

Если сетевого разделения **НЕТ** (система работает штатно), возникает компромисс между задержкой (Latency) и согласованностью (Consistency).

- **PACELC Спецификация:** Если есть **P** (Partition), выбирай между **A** (Availability) и **C** (Consistency); **E**lse (если всё штатно), выбирай между **L**atency (Минимальная задержка) и **C** (Строгая согласованность).
- **Пример CP/EС (MongoDB):** При разделении сети MongoDB изолирует узлы, потерявшие связь с большинством реплик, запрещая туда запись (выбор в пользу строгости **C**). При штатной работе, если настроен параметр записи `w: "majority"`, движок не вернет клиенту статус успеха, пока большинство реплик не запишут данные на диск. Это гарантирует согласованность (**C**), но увеличивает задержку (**L**).
- **Пример AP/EL (Cassandra):** При разделении сети принимает записи на любых живых узлах (выбор доступности **A**). При штатной работе отдает данные мгновенно с локального узла, не дожидаясь репликации на соседей. Задержка минимальна (**L**), но данные на узлах могут временно различаться (согласованность в конечном счете — _Eventual Consistency_).

### 5. Низкоуровневая реализация уровней изоляции транзакций

Для управления конкурентными транзакциями стандартом ANSI SQL определено 4 уровня изоляции. Их реализация под капотом обеспечивается механизмами блокировок (2PL — Two-Phase Locking) или версионированием (MVCC).

#### 1. Read Uncommitted (Чтение незафиксированных данных)

Транзакции изолированы минимально. Допускается феномен **Грязного чтения (Dirty Read)**: Транзакция А может считать строки, которые Транзакция Б изменила, но еще не зафиксировала (`COMMIT`). Если Транзакция Б выполнит `ROLLBACK`, данные Транзакции А станут невалидными.

#### 2. Read Committed (Чтение зафиксированных данных)

Движок гарантирует, что транзакция видит только зафиксированные коммитом данные. Защищает от Грязного чтения.

- Феномен **Неповторяющегося чтения (Non-repeatable Read):** Если Транзакция А считывает строку, затем Транзакция Б изменяет ее и делает `COMMIT`, то повторный вызов `SELECT` внутри Транзакции А вернет уже измененное значение. Под капотом MVCC в PostgreSQL на этом уровне создает моментальный снимок данных (Snapshot) **на каждый отдельный оператор SQL** внутри одной транзакции.

#### 3. Repeatable Read (Повторяемость чтения)

Гарантирует, что повторный запрос данных внутри одной транзакции всегда возвращает идентичный результат. Защищает от Неповторяющегося чтения.

- Под капотом MVCC в PostgreSQL моментальный снимок (Snapshot) генерируется один раз на весь период действия транзакции в момент выполнения первого `SELECT`. Любые коммиты других транзакций, произошедшие параллельно, игнорируются рантаймом.
- Феномен **Фантомного чтения (Phantom Read):** Если Транзакция А делает выборку диапазона строк по условию, а параллельная Транзакция Б вставляет (`INSERT`) новую строку, удовлетворяющую этому условию, то в классических СУБД повторный поиск в Транзакции А вернет «строку-фантом». В PostgreSQL уровень Repeatable Read благодаря MVCC защищает и от фантормных чтений, но в других СУБД (например, MySQL InnoDB) для этого требуются сложные блокировки диапазонов ключей (Next-Key Locks).

#### 4. Serializable (Сериализуемость)

Высший уровень изоляции. Результат параллельного выполнения группы транзакций должен быть полностью эквивалентен их строго последовательному выполнению друг за другом.

- Под капотом PostgreSQL использует алгоритм **SSI (Serializable Snapshot Isolation)**. Блокировок таблиц не происходит. Движок анализирует граф зависимостей между транзакциями на основе совершаемых операций чтения и записи. Если рантайм обнаруживает цикл зависимостей, который может привести к аномалии сериализации (например, _Write Skew_ — коллизия параллельной записи), одна из транзакций принудительно обрывается с ошибкой `40001: serialization_failure`. Приложение обязано отловить эту ошибку и отправить транзакцию на повторное выполнение (Retry).
