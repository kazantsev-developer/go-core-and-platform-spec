## Оркестрация контейнеров в Kubernetes

**Kubernetes (K8s)** — это распределенная модульная платформа оркестрации, предназначенная для автоматизации развертывания, масштабирования и координации изолированных сред исполнения (контейнеров) в рамках вычислительного кластера. Платформа абстрагирует физическое или виртуальное железо хостов, преобразуя его в единый пул ресурсов.

В отличие от автономного Docker-окружения, где единицей планирования является единичный контейнер, в Kubernetes атомарным примитивом рантайма выступает **Под (Pod)**. Под представляет собой логическую группу из одного или нескольких контейнеров, разделяющих общие пространства имен Linux (Network, IPC, UTS) и тома хранения (Volumes). Каждому Поду внутри кластера выделяется индивидуальный, уникальный IP-адрес, что устраняет конфликты связывания портов на одном хосте.

### Главные абстракции и компоненты кластера

Управление инфраструктурой в Kubernetes реализуется через декларативный подход с использованием базовых абстракций:

- **Узел (Node)** — физический сервер или виртуальная машина, выступающая в роли вычислительной единицы. Разделяются на управляющие (Master Node) и рабочие (Worker Node).
- **Деплоймент (Deployment)** — декларативное описание желаемого состояния приложения. Управляет жизненным циклом Подов, задает количество реплик, стратегию обновления и контролирует замену деградировавших контейнеров.
- **Сервис (Service)** — абстракция, определяющая постоянный сетевой адрес (IP или доменное имя) для логической группы Подов. Обеспечивает внутреннюю маршрутизацию и балансировку трафика между репликами, независимо от их перепланирования и изменения их внутренних IP.
- **Ингресс (Ingress)** — входная точка кластера, управляющая внешним доступом к Сервисам (обычно HTTP/HTTPS). Выступает в роли reverse proxy и гейтвея, инкапсулируя правила маршрутизации трафика из внешней сети во внутреннюю сеть кластера.

### Архитектурные отличия рантайма: Standalone Docker против Kubernetes

При деплое приложений через стандартный Docker-Engine (включая утилиту `docker-compose`) управление жизненным циклом ограничено контекстом одного конкретного хоста. При запуске через Docker приложение привязано к локальному окружению, а команды вроде `docker run` или `docker-compose up` выполняются императивно.

В Kubernetes развертывание происходит путем отправки декларативного YAML-манифеста в API-сервер, который самостоятельно определяет оптимальную локацию для запуска. При отказе процесса внутри контейнера Docker использует механизм `restart: always`, который способен поднять упавшее приложение, но бессилен при деградации самого сервера.

Если на физическом сервере происходит критическая аппаратная авария (например, выход из строя материнской платы или блока питания), standalone Docker прекращает функционирование вместе с хостом. Администратор вынужден вручную развертывать конфигурацию на резервном оборудовании и изменять DNS-записи.

Kubernetes реализует модель непрерывного поддержания декларативного состояния через цикл контроля (Control Loop). При потере сетевой связности с узлом или его физическом уничтожении, управляющий слой фиксирует расхождение с манифестом. Оркестратор в автоматическом режиме осуществляет процедуру эвакуации: Поды уничтоженного узла за секунды инициализируются на оставшихся дееспособных серверах кластера без изменения внешних точек входа.

Увеличение количества копий приложения в standalone Docker требует ручного запуска контейнеров на свободных портах хоста и последующей ручной переконфигурации внешнего балансировщика. В Kubernetes масштабирование выполняется декларативно одной командой:

```bash
kubectl scale deployment <deployment-name> --replicas=5
```

Контроллер автоматически разворачивает 5 идентичных копий Пода, распределяет их по узлам кластера с учетом доступных ресурсов и обновляет конфигурацию эндпоинтов соответствующего Сервиса для балансировки входящего трафика.

Статическая привязка сервисов к конкретным виртуальным машинам приводит к фрагментации ресурсов и усложняет управление гетерогенной инфраструктурой. Kubernetes ликвидирует понятие жесткой привязки к хосту. Вся инфраструктура преобразуется в единый пул вычислительных мощностей. Оркестратор самостоятельно распределяет контейнеры по узлам, максимизируя утилизацию процессора и памяти. Логическое разграничение сред (Production / Development) достигается внутри одного кластера с помощью виртуальной изоляции пространств имен (Namespaces).

### Инфраструктурная оптимизация хоста

Развертывание кластера поверх систем виртуализации оптимизирует аллокацию аппаратных ресурсов хоста по двум ключевым направлениям:

#### 1. Утилизация дискового пространства

При статической нарезке виртуальных машин под каждую логическую задачу администратор вынужден развертывать изолированные экземпляры гостевой операционной системы. Каждая ВМ резервирует фиксированный объем дискового пространства под системный корень, логи и утилиты, что дублирует бинарные файлы ОС на накопителе. В инфраструктуре Kubernetes количество управляющих виртуальных машин сокращается до минимума. Логическая изоляция приложений в пространствах имен в рамках общего дискового пространства нод устраняет системный оверхед.

#### 2. Распределение оперативной памяти

Статическое выделение оперативной памяти (RAM) для виртуальных машин жестко фиксирует ресурсы хоста. Свободная память простаивающей ВМ разработки не может быть утилизирована нагруженным продакшн-сервисом без остановки машины и ручного изменения конфигурации гипервизора. Kubernetes объединяет физическую память всех подключенных узлов в гомогенный пул. Рантайм динамически распределяет память между подами на основе механизмов гарантированного запроса (`requests`) и жесткого ограничения (`limits`), перенаправляя ресурсы в режиме реального времени туда, где фиксируется пиковая транзакционная нагрузка.

### Механика обновления без остановки сервиса (Zero-Downtime Rolling Update)

Стандартный механизм обновления в standalone Docker принудительно останавливает текущий контейнер для освобождения сетевого порта хоста, создавая окно недоступности сервиса (Downtime). Запуск двух копий одного контейнера на одном хосте напрямую невозможен из-за конфликта связывания портов (`bind: address already in use`).

Kubernetes использует сетевой плагин CNI для выделения каждому Поду уникального IP-адреса, что устраняет конфликты портов на физическом интерфейсе ноды.

#### Алгоритм выполнения RollingUpdate:

1. Контроллер Deployment фиксирует изменение манифеста приложения (новый тег образа).
2. Рантайм инициализирует новый Под с обновленной версией кода параллельно со старым Подом на доступных ресурсах памяти хоста.
3. Внутренний балансировщик (Service) продолжает направлять входящий трафик на старый Под.
4. Запускается механизм валидации готовности (**Readiness Probe**). Kubernetes выполняет низкоуровневые проверки (HTTP-запросы или TCP-пинг внутреннего порта контейнера), ожидая полной готовности приложения к обработке запросов.
5. После перехода нового Пода в статус `Ready` балансировщик трафика мгновенно переключает сетевые пакеты клиентов на IP-адрес нового Пода.
6. Старый Под переводится в статус `Terminating`, получает сигнал `SIGTERM` для корректного завершения активных сессий (Graceful Shutdown) и уничтожается.

### Системные спецификации: Сравнение Vanilla K8s и K3s

**K3s** — это официально сертифицированный дистрибутив Kubernetes, полностью удовлетворяющий спецификации Kubernetes API, но оптимизированный для сред с жесткими ресурсными ограничениями. В отличие от него, дистрибутивы вроде MicroK8s ориентированы на локальную разработку и используют другие механизмы упаковки зависимостей (например, snap-пакеты).

#### 1. Архитектура слоя хранения (Storage Engine Core)

Классический Kubernetes (Vanilla K8s) использует в качестве распределенного хранилища конфигураций базу данных **etcd**. Она функционирует на базе алгоритма консенсуса **Raft**, требуя строго нечетного количества узлов (минимум 3) для формирования кворума. При падении одного узла в кластере из двух нод кворум теряется, и etcd полностью блокирует операции записи во избежание состояния разделения сети (_Split-Brain_). Движок etcd критичен к задержкам дисковой подсистемы (IOPS) и потребляет значительный объем RAM.

Дистрибутив K3s для малых конфигураций (1–2 ноды) заменяет etcd на встроенную легковесную СУБД **SQLite**. Для трансляции вызовов Kubernetes API в SQL-команды задействован архитектурный программный слой **Kine** (Kine is not etcd). Это устраняет необходимость содержания нечетного количества серверов для кворума и минимизирует утилизацию оперативной памяти управляющего узла (Master Node). При масштабировании Kine позволяет переключить бэкенд на внешние реляционные базы (PostgreSQL, MySQL).

#### 2. Критерии выбора СУБД для реестра кластера

Рантайм оркестратора накладывает жесткие требования к транзакционности (поддержка ACID). По этой причине для хранения состояния кластера невозможно использовать неперсистентные In-Memory решения (Redis) или NoSQL системы (MongoDB). Хранилищу требуется обеспечить строгое соблюдение изоляции транзакций и атомарности записи. Это исключает возникновение состояния гонки (Race Condition) при одновременной модификации сетевых политик или манифестов подов несколькими независимыми контроллерами.

#### 3. Монолитная сборка зависимостей

Классический K8s поставляется в виде набора разрозненных демонов (`kube-apiserver`, `kube-scheduler`, `kube-controller-manager`, `kubelet`), требующих ручной координации и самостоятельной установки сетевого плагина CNI и контроллера Ingress.

K3s скомпилирован в единый бинарный файл размером менее 70 МБ. Все необходимые компоненты (сетевой плагин Flannel, TLS-роутер Traefik, локальный балансировщик) интегрированы непосредственно внутрь исполняемого файла. Из кодовой базы полностью вырезаны устаревшие драйверы внешних облачных провайдеров, что снижает требования к ресурсам процессора и диска.

### Низкоуровневое устройство подсистемы хранения etcd

База данных etcd — это высоконадежное распределенное Key-Value хранилище, разработанное в 2013 году компанией CoreOS. Архитектурно название представляет собой концепт распределенной директории конфигураций Linux (`/etc distributed`). В инфраструктуре оркестратора etcd выполняет роль единого источника правды (Single Source of Truth), фиксируя структуру всех ресурсов кластера.

Данные организованы в виде иерархического пространства ключей без использования реляционных таблиц. Для координации состояния узлов etcd использует алгоритм консенсуса Raft, работающий по принципу выделенного Лидера (Leader Election). Изменения конфигурации применяются только после подтверждения записи на диск со стороны большинства узлов — кворума, рассчитываемого математически как общее количество узлов, деленное на два, плюс один.

Строго нечетный базис (3, 5 или 7 нод) необходим для однозначного формирования большинства при возникновении сетевого разделения хостов. При фиксации транзакций etcd опирается на принципы ACID:

- **Атомарность (Atomicity):** Операция изменения конфигурации ресурса (например, создание Пода) применяется полностью. Частичная запись или повреждение манифеста при сбое питания исключены: рантайм полностью откатывает транзакцию к исходному состоянию.
- **Согласованность (Consistency):** Алгоритм гарантирует линейную последовательность транзакций. Любой легитимный запрос на чтение из etcd возвращает идентичное и актуальное состояние системных конфигураций на всех узлах кластера.
- **Изоляция (Isolation):** Одновременные конкурентные запросы на изменение объектов кластера изолируются базой данных. Исключается возникновение логических конфликтов или состояний гонки (Race Condition) при параллельном деплое из разных источников.
- **Стойкость (Durability):** После подтверждения коммита данные гарантированно фиксируются в WAL (Write-Ahead Log) файле и моментальном снимке (Snapshot) на энергонезависимом накопителе, что предотвращает их потерю.
