## Спецификация наблюдаемости распределенных систем (Observability): Метрики, Логирование и Распределенная трассировка

В микросервисных архитектурах высокой пропускной способности (Highload) и распределенных кластерах Kubernetes классический мониторинг доступности узлов (работает/не работает) является неэффективным. Концепция наблюдаемости (Observability) декларирует способность системы предоставлять полную телеметрию внутреннего состояния через три фундаментальных столпа: Метрики (Metrics), Структурированные логи (Logs) и Трассировки (Traces). Это позволяет оперативно локализовать деградацию производительности, сетевые аномалии и каскадные сбои на любых уровнях инфраструктуры.

---

### 1. Сбор и агрегация метрик: Архитектура Prometheus и Grafana

Метрики предоставляют агрегированную цифровую информацию о поведении системы за определенный промежуток времени. Они оптимизированы для хранения временных рядов (Time-Series) и запуска систем мгновенного оповещения (Alerting).

#### Модель сбора данных: Pull против Push

Prometheus кардинально отличается от классических систем сбора метрик использованием **Pull-модели**.

- Брокер Prometheus самостоятельно инициирует HTTP GET-запросы по расписанию (Scrape Interval, например, каждые 15 секунд) на специализированные эндпоинты микросервисов (обычно `/metrics`). Приложения обязаны отдавать метрики в текстовом формате OpenMetrics.
- Это полностью изолирует бэкенд от инфраструктуры мониторинга: если сервер Prometheus начнет деградировать или упадет под высокой нагрузкой, он **физически не сможет заблокировать или затормозить работу боевых Go-микросервисов**.
- Для короткоживущих задач (Cron-скрипты, K8s Jobs), которые завершаются быстрее, чем наступит цикл опроса, применяется промежуточный буфер — **Pushgateway**, куда скрипты отправляют данные по Push-модели, а Prometheus забирает их стандартным Pull-методом.

#### Низкоуровневые типы данных в памяти Prometheus

Рантайм Go через официальный пакет `://github.com` оперирует четырьмя базовыми типами метрик:

1. **Counter (Счетчик):** Кумулятивная метрика, которая может только монотонно увеличиваться (или сбрасываться в ноль при перезапуске процесса). Применяется для подсчета общего количества обработанных HTTP-запросов, системных ошибок или отправленных байт.
2. **Gauge (Измеритель):** Метрика, отражающая мгновенное значение, которое может как увеличиваться, так и уменьшаться. Используется для мониторинга текущей утилизации оперативной памяти (RAM Heap), загрузки CPU, количества активных горутин в планировщике или занятых сокетов в пуле соединений `pgx/v5`.
3. **Histogram (Гистограмма):** Сложнейший тип для Highload-систем. Она считает количество событий, попавших в заранее определенные диапазоны значений (Buckets), и общую сумму всех значений. Незаменима для расчета латентности (Latency) запросов. На основе гистограмм Prometheus вычисляет точные квантили производительности (p95, p99) через математическую функцию `histogram_quantile()`.
4. **Summary (Сводка):** Рассчитывает квантили непосредственно на стороне самого Go-приложения за скользящее временное окно. Обладает более точными показателями для единичного сервиса, но математически не подлежит агрегации между несколькими репликами подов в Kubernetes, в отличие от Histogram.

### 2. Централизованное логирование: Стек Loki и структурированный JSON

Логи отражают дискретные текстовые события с детальным контекстом происходящего. В высоконагруженных распределенных системах запись логов в виде неструктурированного текста (Plain Text) в файлы — это критическая архитектурная ошибка.

#### Требование к Структурированному логированию (Structured Logging)

Каждая строчка лога в Go-микросервисе (через пакеты `slog`, `zap` или `logrus`) обязана компилироваться строго в бинарный формат **JSON**. Это превращает лог из нечитаемой строки в индексируемый объект с четкими полями: `time`, `level`, `component`, `trace_id`, `message`, `error_details`. Инфраструктурные сборщики могут парсить такие логи без регулярных выражений, моментально фильтруя ошибки по конкретным пользователям или методам API.

#### Архитектура Grafana Loki (Сжатие логов и отказ от полнотекстового индекса)

В отличие от тяжелого стека Elasticsearch, который строит гигантские инвертированные индексы под каждое слово в тексте лога (что раздувает дисковое хранилище в разы и перегружает RAM процессора), **Grafana Loki** спроектирована по принципу «Dumb Index / Smart Query».

- Loki индексирует исключительно **метаданные** лога — метки и теги (Labels), которые полностью совпадают с метками Prometheus (например, `pod=mentorship-backend-pod`, `env=production`).
- Сам текст JSON-логов сжимается в компактные бинарные чанки (Chunks) и сбрасывается в дешевое S3-совместимое объектное хранилище (MinIO / AWS S3) последовательным методом (Sequential I/O).
- Поиск по тексту выполняется налету через мощный движок запросов LogQL силами воркеров Loki при чтении чанков, что сокращает расходы на инфраструктуру логирования на 70–80%.

### 3. Распределенная трассировка (Distributed Tracing): Спецификация OpenTelemetry и Jaeger

Когда один пользовательский клик на фронтенде порождает каскадную цепочку вызовов через 10 независимых изолированных микросервисов (Next.js -> gRPC -> Go API -> Redis -> Kafka -> Worker -> PostgreSQL), локализовать причину деградации латентности без трассировки невозможно. **Distributed Tracing** позволяет визуализировать полный путь прохождения единичного запроса сквозь весь распределенный ландшафт.

#### Механика Context Propagation (Распространение контекста по сети)

В основе распределенной трассировки лежит промышленный стандарт **W3C Trace Context**:

1. На входных воротах архитектуры (Nginx Reverse Proxy или API Gateway) для нового входящего HTTP-запроса рантайм генерирует глобальный уникальный 64-битный идентификатор трассировки — **Trace ID**.
2. В рамках этой трассировки каждый шаг внутри конкретного микросервиса называется **Спаном (Span)**, который имеет свой `Span ID` и ссылку на родительский `Parent Span ID`.
3. При межсервисном взаимодействии (HTTP/gRPC вызовы) клиент Обязан выполнить процедуру **Inject (Внедрение)**: упаковать текущие `Trace ID` и `Span ID` в стандартизированные сетевые заголовки `traceparent` (например, `traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01`).
4. Принимающий микросервис выполняет процедуру **Extract (Извлечение)**: считывает заголовок из сети, парсит его, восстанавливает Trace ID и порождает у себя внутри локального контекста `context.Context` новый дочерний Спан.

#### Сбор и визуализация данных в Jaeger

Каждое Go-приложение асинхронно (в фоновом потоке, чтобы не создавать задержек для бизнес-логики) отправляет сформированные бинарные спаны на агент или коллектор **OpenTelemetry (OTEL)**. Коллектор агрегирует данные и передает их в хранилище **Jaeger**. Инженер в интерфейсе Jaeger видит сквозной древовидный граф исполнения запроса с точностью до наносекунды на каждый метод, что позволяет моментально определить, на каком конкретно gRPC-вызове или SQL-запросе застрял пользователь.
