## Спецификация межсервисного взаимодействия: gRPC и механизмы сериализации Protocol Buffers

В распределенных микросервисных архитектурах высокой пропускной способности (Highload) классический подход REST API на базе передачи текстового JSON поверх HTTP/1.1 порождает критический оверхед на сериализацию, парсинг строк и нерациональное использование сетевых сокетов. gRPC (Google Remote Procedure Call) является промышленным стандартом межсервисного взаимодействия, обеспечивающим минимальную задержку (Latency), бинарную компактность трафика и строгую типизацию контрактов.

---

### 1. Транспортный слой gRPC: Архитектурная зависимость от HTTP/2

gRPC принципиально не может функционировать поверх HTTP/1.1, так как его внутренняя логика обработки вызовов опирается на низкоуровневые возможности бинарного протокола HTTP/2.

#### Мультиплексирование и утилизация сокетов

Вместо постоянного открытия и закрытия TCP-соединений или удержания пула тяжелых сокетов, gRPC-клиенты открывают **один персистентный TCP-канал (Connection)** к микросервису. Внутри этого единого сокета рантайм мультиплексирует тысячи независимых виртуальных потоков (Streams). Каждый отдельный RPC-вызов выполняется в своем изолированном Stream, что полностью ликвидирует накладные расходы на сетевое рукопожатие (Handshake) для последующих запросов.

#### Механика передачи метаданных через фреймы HTTP/2

gRPC полностью инкапсулирует вызовы процедур в стандартные бинарные фреймы HTTP/2:

- **Инициализация вызова (Заголовки):** Клиент отправляет фрейм `HEADERS`. gRPC передает имя вызываемого метода в системном pseudo-header `:path` (например, `/:path: /pb.UserService/GetUser`). Тип контента жестко фиксируется в заголовке `content-type: application/grpc`. Авторизационные токены и контекстные метаданные передаются в этом же фрейме в виде бинарных метаданных (`metadata`).
- **Полезная нагрузка (Payload):** Тело запроса и ответа передается во фреймах `DATA`. Каждый фрейм `DATA` в gRPC снабжается 5-байтным префиксом (1 байт — флаг сжатия, 4 байта — длина сообщения), за которым следует чистый бинарный поток сериализованного Protocol Buffers.
- **Завершение вызова (Трейлеры):** Статус завершения процедуры gRPC передает не в стандартном HTTP-статусе (он всегда возвращает `200 OK`, если транспорт жив), а в финальном фрейме `HEADERS` с установленным флагом `END_STREAM`. Этот фрейм содержит HTTP-трейлеры (Traefik/Nginx умеют их проксировать), где передаются системный статус-код gRPC (`grpc-status: 0` для успехов, или коды ошибок `1-16`) и текстовое сообщение об ошибке (`grpc-message`).

### 2. Бинарная сериализация Protocol Buffers (Protobuf)

**Protocol Buffers** — это машинно-независимый, нейтральный к языкам программирования механизм сериализации структурированных данных. В отличие от JSON, который хранит данные в виде человекочитаемого текста вместе с именами полей, Protobuf компилирует данные в плотный бинарный поток, где имена полей отсутствуют, а валидация структуры происходит на основе числовых тегов.

#### Низкоуровневое кодирование: TLV и Varints

Protobuf оптимизирует каждый байт на уровне регистров процессора, используя кодирование **TLV (Tag-Length-Value)** и схему избыточной упаковки целых чисел **Varints (Variable-Length Quantities)**.

- **Концепция Varints:** Позволяет упаковывать целые числа (`int32`, `int64`) в переменное количество байт. Вместо того чтобы всегда занимать 8 байт на диске или в сети под число `42` типа `int64`, Varints задействует старший бит каждого байта (Most Significant Bit — MSB) как маркер: если MSB равен `1`, значит, следующий байт тоже относится к этому числу; если MSB равен `0`, число завершено. Мелкие числа упаковываются всего в 1 байт.
- **Отсутствие имен полей:** В бинарном потоке Protobuf поле идентифицируется исключительно его уникальным номером (тегом), заданным в `.proto` файле. Поток состоит из пар: `[Номер поля + Тип данных (Wire Type)] -> [Значение]`. Это сокращает объем передаваемых данных по сети на 60–80% по сравнению с JSON и полностью исключает накладные расходы на парсинг строк парсером.

### 3. Режимы стриминга в gRPC

gRPC предоставляет 4 фундаментальных режима взаимодействия между распределенными компонентами системы:

1. **Унарный (Unary RPC):** Классический запрос-ответ. Клиент отправляет один фрейм `HEADERS` и один `DATA`, сервер отвечает аналогично. Применяется для стандартных CRUD операций.
2. **Серверный стриминг (Server Streaming RPC):** Клиент отправляет один запрос, а сервер открывает долгоживущий поток данных и последовательно отправляет множество фреймов `DATA` (например, стриминг логов из контейнеров, тяжелые выборки из БД, Real-Time котировки). Поток закрывается сервером через финальный фрейм трейлеров.
3. **Клиентский стриминг (Client Streaming RPC):** Клиент открывает поток и непрерывно загружает пачки фреймов `DATA` на сервер (например, загрузка больших blob-файлов или чанков видео частями). По завершении клиент закрывает поток, и сервер возвращает один унарный ответ.
4. **Двунаправленный стриминг (Bidirectional Streaming RPC):** Полнодуплексный режим. Клиент и сервер одновременно открывают два независимых потока в рамках одного HTTP/2 Stream и могут асинхронно отправлять друг другу сообщения без жесткой последовательности «запрос-ответ». Идеально для real-time чатов и игровых движков.

### 4. Интеграция с примитивами рантайма Go

Написание gRPC-сервисов на языке Go нативно интегрируется с ключевыми компонентами рантайма, обеспечивая высокую отказоустойчивость:

- **Управление конкурентностью:** Под капотом официальный gRPC-сервер (`google.golang.org/grpc`) на каждый входящий сетевой Stream автоматически выделяет изолированную горутину. Потокобезопасность обеспечивается тем, что рантайм gRPC берет на себя управление блокировками при записи во фреймы сокета.
- **Транзит контекстов и таймаутов:** gRPC нативно поддерживает `context.Context`. Если клиент вызывает метод с контекстом, у которого задан таймаут (`context.WithTimeout`), gRPC автоматически сериализует время жизни запроса в HTTP/2 заголовок `grpc-timeout`. Принимающий микросервер считывает этот заголовок и автоматически порождает у себя дочерний контекст с точно таким же оставшимся временем жизни. Если время истекает, вся цепочка микросервисов мгновенно прекращает вычисления, высвобождая ресурсы процессора и потоки горутин.
