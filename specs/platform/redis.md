## Спецификация Redis: In-Memory архитектура, структуры данных и механизмы обеспечения персистентности

В высоконагруженных распределенных системах (Highload) дисковые СУБД (PostgreSQL, MongoDB) не способны обеспечить субмиллисекундное время ответа (Latency) при интенсивном потоке запросов на чтение. Redis (Remote Dictionary Server) решает эту задачу, функционируя как высокопроизводительное хранилище структур данных в оперативной памяти (In-Memory). Архитектурная стабильность Redis базируется на неблокирующем вводе-выводе, однопоточном ядре вычислений и специализированных бинарных представлениях объектов в RAM.

---

### 1. Однопоточная архитектура: Цикл обработки событий и неблокирующий I/O

Главная особенность Redis — его сетевое ядро и обработка команд выполняются в **одном основном системном потоке**.

#### Устранение накладных расходов на конкурентность

Сознательный отказ от многопоточности внутри ядра вычислений ликвидирует критические оверхеды, характерные для классических СУБД:

- Отсутствие затрат процессора на контекстное переключение потоков ОС (Context Switching).
- Полное отсутствие коллизий, гонок данных (Race Conditions) и накладных расходов на синхронизацию (Mutexes, Read-Write Locks).
- Гарантия абсолютной атомарности выполнения любой единичной команды по умолчанию.

#### Механизм мультиплексирования сокетов (Event Loop)

Для одновременного удержания сотен тысяч открытых сетевых соединений Redis использует системные механизмы мультиплексирования ввода-вывода операционной системы — **epoll** в Linux (или `kqueue` в BSD).

Основной поток рантайма крутится в бесконечном цикле обработки событий (**Event Loop**). Он асинхронно опрашивает сетевые сокеты через `epoll`. Как только в сокет от Go-клиента прилетают байты команды, ядро ОС генерирует событие, Redis мгновенно считывает его, выполняет команду в памяти и отправляет результат обратно. Если команда тяжелая (например, `KEYS *` на базе в миллион элементов), она полностью заблокирует единственный поток вычислений, и все остальные клиенты встанут в мертвую очередь по таймауту.

### 2. Низкоуровневые структуры данных в оперативной памяти

Redis хранит ключи и значения не в виде сырого текста, а в виде строго оптимизированных бинарных структур данных. Каждому прикладному типу данных под капотом соответствует внутреннее эффективное кодирование (Encoding).

- **Strings (Строки):** Реализованы через структуру **SDS (Simple Dynamic String)**. В отличие от классического C-строкового типа, SDS хранит длину строки в отдельном заголовке. Это позволяет рантайму возвращать длину за константное время $O(1)$ вместо полного перебора символов за $O(N)$ и предотвращает переполнение буфера.
- **Hashes (Хэш-таблицы) и Sets (Множества):** При малом объеме данных кодируются как плотный, последовательный в памяти массив **ZipList** для утилизации кэш-линий процессора (Cache Locality). При разрастании структуры Redis прозрачно преобразует их в полноценную хэш-таблицу **Dict**, использующую прогрессивное рехэширование (две таблицы, постепенный перенос бакетов во время входящих запросов), аналогично `hmap` в Go.
- **Sorted Sets (Сортированные множества):** Уникальная структура, позволяющая извлекать диапазоны элементов, отсортированных по весу (Score). Реализована через комбинацию хэш-таблицы (для быстрого поиска веса по ключу за $O(1)$) и **Списка пропусков (Skip List)**. Skip List — это вероятностная многоуровневая структура из связанных списков, позволяющая осуществлять поиск, вставку и удаление элементов за логарифмическое время $O(\log N)$, являясь более простой и низкоаллокационной альтернативой сбалансированным B-Tree деревьям.

### 3. Механики персистентности: RDB против AOF

Поскольку оперативная память энергозависима, для предотвращения полной потери данных при сбое питания или перезагрузке демона Redis использует два взаимодополняющих механизма создания дампов на диск хоста.

#### RDB (Redis Database Backup — Точечные снимки)

Механизм создания компактных бинарных моментальных снимков всей базы данных на диске по расписанию (например, каждые 15 минут, если изменилось более 100 ключей).

- **Системный вызов `fork()` и оптимизация Copy-on-Write (COW):** Чтобы процесс записи гигабайтов данных на медленный диск не блокировал единственный рабочий поток вычислений, Redis вызывает системную команду ядра Linux `fork()`. ОС порождает точную копию процесса — дочерний процесс (Child Process), который имеет доступ к тем же страницам оперативной памяти.
- На уровне ядра Linux включается механизм **Copy-on-Write**. Дочерний процесс последовательно считывает страницы памяти и пишет их в бинарный файл RDB на диск. Основной поток в этот момент продолжает принимать запросы от приложений. Если в процессе записи дампа Go-клиент изменяет какой-то ключ, ядро Linux физически копирует конкретную 4-килобайтную страницу памяти в новое место, изолируя данные для дочернего процесса. RDB практически не создает оверхеда на производительность, но при аварии данные, созданные после последнего снимка, будут безвозвратно потеряны.

#### AOF (Append Only File — Лог дозаписи операций)

Максимально надежный режим. Redis записывает каждую входящую команду модификации данных (`SET`, `DEL`, `HSET`) в виде текстового лога в конец файла на диске последовательным методом (Sequential I/O).

- **Режимы синхронизации `fsync`:** Запись сначала попадает в буфер ОС. Частота сброса буфера на диск регулируется параметром `appendfsync`:
  - `always` — принудительный `fsync` диска на каждую операцию. Гарантия надежности, но производительность падает до уровня классических дисковых СУБД.
  - `everysec` — фоновый `fsync` каждую секунду. Промышленный баланс: максимальная скорость работы In-Memory с риском потерять максимум 1 секунду данных при крахе по питанию.
- **Фоновая оптимизация (AOF Rewrite):** Со временем файл AOF раздувается (если один ключ обновили миллион раз, в файле будет миллион строк). Рантайм запускает фоновый процесс `BGREWRITEAOF`, который через `fork()` считывает текущий стейт памяти и создает новый сжатый файл AOF, содержащий только финальные минимально необходимые команды для восстановления актуального состояния.

### 4. Стратегии вытеснения данных (Eviction Policies)

Когда объем данных в Redis достигает жесткого физического лимита оперативной памяти, настроенного в параметре `maxmemory`, брокер начинает вытеснять старые ключи для освобождения места под новые записи.

- **noeviction:** Стратегия по умолчанию. При достижении лимита памяти Redis на любые попытки записи (`SET`, `HSET`) возвращает клиенту системную ошибку `OOM command not allowed когда maxmemory exceeded`. Применяется, если Redis используется как критическая основная база данных.
- **volatile-lru / allkeys-lru (Least Recently Used):** Вытесняет ключи, которые запрашивались **дольше всего по времени**. Redis не хранит точный хронологический список использования всех ключей (это создало бы гигантский оверхед на память). Вместо этого применяется вероятностный алгоритм: Redis берет случайную выборку ключей (например, 5 штук) и уничтожает из них тот, у которого системный маркер `lru` внутри структуры объекта является самым старым.
- **volatile-lfu / allkeys-lfu (Least Frequently Used):** Вытесняет ключи, которые запрашивались **меньше всего по количеству раз (по частоте)** за единицу времени. Алгоритм защищает систему от ложных вытеснений полезных данных при случайных единичных сканированиях базы.
