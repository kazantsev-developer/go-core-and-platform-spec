## Ключевое слово defer в Go

**`defer`** — это встроенное ключевое слово Go, предназначенное для отложенного вызова функции. Отложенная функция гарантированно выполняется непосредственно перед выходом из окружающей её функции — после выполнения основного тела и оператора `return`, но до того, как управление вернется вызывающей стороне.

Главное назначение `defer` — гарантированное освобождение ресурсов (файлы, сокеты, мьютексы) и обработка паник вне зависимости от того, как завершилась функция (штатно, через `early return` или аварийно).

---

### Внутреннее устройство (Под капотом)

Начиная с Go 1.14, рантайм использует три механизма оптимизации `defer` (выбор зависит от компилятора):

1. **Open-coded defers (Открытый код):** Если количество деферов фиксировано и нет циклов, компилятор встраивает код отложенных функций напрямую в точки выхода из родительской функции. Оверхед равен нулю (работает так же быстро, как обычный вызов функции).
2. **Stack-allocated defers (Аллокация на стеке):** Если открытый код неприменим, структура `_defer` выделяется на стеке горутины, что минимизирует работу с кучей (Garbage Collector).
3. **Heap-allocated defers (Аллокация в куче):** Используется в циклах `for`. Структура `_defer` выделяется в куче, связываясь в односвязный список внутри структуры горутины (`g._defer`). Это самый медленный вариант.

---

### 4 жестких правила defer, которые нужно знать наизусть

#### 1. Порядок выполнения: LIFO (Стек)

Если в одной функции объявлено несколько `defer`, они складываются в стек горутины и выполняются в **обратном порядке** их объявления (Last In, First Out — «последним пришел, первым ушел»).

#### 2. Момент вычисления аргументов функции

Аргументы отложенной функции вычисляются **в момент выполнения строки с ключевым словом `defer`**, а не в момент фактического вызова самой функции на выходе. Значения копируются сразу.

#### 3. Момент вычисления блока внутри анонимного дефера (Замыкания)

Если внутри `defer` используется анонимная функция без аргументов (замыкание), то переменные из внешней функции оцениваются **непосредственно в момент выполнения дефера** (на выходе из функции), так как функция ссылается на указатель переменной, а не на её скопированное значение.

#### 4. Взаимодействие с именованным возвращаемым значением (`named return`)

`defer` может напрямую перезаписать значение, которое возвращает функция, если в сигнатуре функции используются именованные возвращаемые переменные. Это происходит потому, что `return` сначала присваивает значения возвращаемым переменным, затем запускаются деферы, и только потом происходит физический выход (`RET`).

### Примеры кода со всеми подвохами

#### 1. Разница между вычислением аргумента и замыканием

```go
package main

import "fmt"

func main() {
	x := 10

	// Правило 2: Аргумент копируется СЕЙЧАС. На выходе напечатает 10.
	defer fmt.Println("Вызов с аргументом x:", x)

	// Правило 3: Замыкание захватывает ссылку на x. На выходе напечатает 20.
	defer func() {
		fmt.Println("Вызов через замыкание x:", x)
	}()

	x = 20
	fmt.Println("Основное тело функции, x =", x)
}
/*
Вывод программы:
Основное тело функции, x = 20
Вызов через замыкание x: 20
Вызов с аргументом x: 10
*/
```

#### 2. Изменение именованного возвращаемого значения (`named return`)

```go
package main

import "fmt"

func getValues() (a int, b int) {
	a = 1
	b = 1

	// Дефер перехватывает переменные «a» и «b» до физического выхода из функции
	defer func() {
		a += 10 // Напрямую меняет именованную переменную
	}()

	return a, b // Шаг 1: a = 1, b = 1; Шаг 2: Вызов дефера (a становится 11); Шаг 3: Выход
}

func main() {
	x, y := getValues()
	fmt.Printf("x: %d, y: %d\n", x, y) // Вывод: x: 11, y: 1
}
```

#### 3. Критическая утечка памяти/дескрипторов в цикле `for`

```go
package main

import "os"

func badPractice() {
	for i := 0; i < 1000; i++ {
		f, _ := os.Open("log.txt")
		// ПЛОХО: Дефер не выполнится на каждой итерации цикла!
		// Стек деферов будет расти в куче, а файлы останутся открытыми, пока не завершится ВСЯ функция badPractice.
		defer f.Close()
	}
}

func goodPractice() {
	for i := 0; i < 1000; i++ {
		// ПРАВИЛЬНО: Выносим логику в анонимную или под-функцию.
		// Дефер срабатывает сразу при выходе из итерации (границы функции)
		func() {
			f, _ := os.Open("log.txt")
			defer f.Close()
		}()
	}
}
```

#### 4. Механика Panic / Recover

`recover()` работает **только** внутри отложенных функций. Если вызвать его в основном теле программы, он просто вернет `nil` и ничего не перехватит.

```go
package main

import "fmt"

func causePanic() {
	defer func() {
		if r := recover(); r != nil {
			fmt.Printf("Паника успешно локализована: %v\n", r)
		}
	}()

	fmt.Println("Выполняем опасный код...")
	panic("критическая ошибка базы данных")
	fmt.Println("Этот код никогда не выполнится")
}

func main() {
	causePanic()
	fmt.Println("Программа продолжает стабильную работу после паники.")
}
```

### Резюме по оптимизации

В продакшене `defer` безопасен и практически не имеет накладных расходов из-за open-coded оптимизации. Единственное жесткое табу: **никогда не использовать `defer` в длинных или бесконечных циклах `for`**, чтобы избежать раздувания памяти (Heap-allocated defers) и исчерпания системных лимитов на открытые файлы/соединения.
