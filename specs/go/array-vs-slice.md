## Разница между массивами и слайсами в Go

Массив и слайс имеют принципиальные различия на уровне системы типов, низкоуровневого управления памятью и механики обработки компилятором.

---

### Аллокация и поведение памяти

**Массив (`[N]T`)** — это цельный, непрерывный блок памяти фиксированного объема. Его размерность детерминирована на этапе компиляции. При объявлении внутри локальной области видимости (внутри функции) компилятор практически всегда выделяет под него память на стеке горутины через механизм escape-анализа. Это гарантирует нулевой оверхед и отсутствие работы для Сборщика мусора (GC).

**Слайс (`[]T`)** — это заголовок (дескриптор структуры `runtime.slice`), состоящий из трех полей: указатель на область памяти, длина (`len`) и емкость (`cap`). Сам заголовок (24 байта на x64) всегда создается на стеке. Однако нижележащий массив данных, на который он ссылается, при динамическом расширении (`append`) или выходе из области видимости функции выносится компилятором в кучу (heap), генерируя аллокации и нагрузку на GC.

---

### Система типов и сравниваемость

Массив является полноценным сравниваемым типом (`comparable`), если тип его элементов поддерживает операторы сравнения (`==`, `!=`). Размер массива входит в определение его типа. Это означает, что типы `[3]int` и `[4]int` абсолютно несовместимы, их нельзя приводить друг к другу или сравнивать между собой. Из-за поддержки интерфейса сравнимости массивы разрешено использовать в качестве ключей в хэш-таблицах (`map`).

Слайс не имеет фиксированного размера в типе, поэтому `[]int` всегда остается одним и тем же типом. Слайсы запрещено сравнивать друг с другом через операторы `==` или `!=` (компилятор выдаст ошибку, разрешено только сопоставление с `nil`). Соответственно, слайс принципиально невозможно использовать как ключ в `map`.

---

### Механика копирования при передаче в функции

В Go отсутствует передача по ссылке, все аргументы копируются по значению.

При передаче массива в функцию компилятор генерирует код, который физически **копирует весь массив поэлементно**. Передача массива большой размерности (например, `[10000]int`) приводит к немедленному забиванию стека дублирующимися данными и просадке производительности CPU. Любая модификация элементов внутри функции работает с локальной копией и не меняет исходный массив.

При передаче слайса копируются только 24 байта его заголовка. Так как адрес указателя на нижележащий массив в копии остается прежним, функция получает прямой доступ к исходным данным. Изменение элементов слайса внутри функции мгновенно отразится на исходных данных снаружи.

---

### Примеры кода и крайние случаи

#### Изменение данных: Полная изоляция массива против общего массива слайса

```go
package main

import "fmt"

func modifyArray(arr [3]int) {
	arr[0] = 999 // Модификация локального стека функции
}

func modifySlice(sl []int) {
	sl[0] = 999 // Модификация памяти по исходному указателю
}

func main() {
	a := [3]int{1, 2, 3}
	modifyArray(a)
	fmt.Println(a) // Вывод: [1 2 3] (исходный массив не изменился)

	s := []int{1, 2, 3}
	modifySlice(s)
	fmt.Println(s) // Вывод: [999 2 3] (данные изменены)
}
```

#### Использование массива в качестве ключа хэш-таблицы

```go
package main

import "fmt"

func main() {
	// Массивы фиксированной длины сравнимы, код валиден
	m := make(map[2]int]string)

	m[[2]int{55, 66}] = "Node_A"
	m[[2]int{77, 88}] = "Node_B"

	fmt.Println(m[[2]int{55, 66}]) // Вывод: Node_A

	// Попытка использовать []int вместо [2]int вызовет ошибку компиляции:
	// invalid map key type []int
}
```

### Системная оптимизация

Массивы эффективны в узких сценариях, когда объем коллекции жестко зафиксирован бизнес-логикой на этапе написания кода (например, байтовые маски, UUID, хэши фиксированной длины). Во всех остальных инженерных кейсах стандартом является слайс. Для предотвращения множественных аллокаций памяти и копирования данных при вызовах `append`, слайсы необходимо инициализировать с явным указанием емкости через `make([]T, 0, capacity)`.
