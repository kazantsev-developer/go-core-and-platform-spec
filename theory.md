У Go есть несколько архитектурных преимуществ. Во-первых, отсутствие единого реестра пакетов — зависимости тянутся прямо из Git, что снижает риски блокировки. Во-вторых, система `vendor` и `go.sum` обеспечивают воспроизводимость сборки и криптографическую проверку целостности — это критично для безопасности в корпоративной среде. В-третьих, стандартная библиотека покрывает большинство задач бэкенда, что уменьшает количество внешних зависимочей. Это контрастирует с npm, где даже простой проект может тянуть сотни пакетов, каждый из которых — потенциальный вектор атаки

Checksum / Hash
**Контрольная сумма** — это уникальная «цифровая подпись» файла, которая получается с помощью математической формулы (хэш-функции). Номер отслеживания файла.

Контрольные суммы — это криптографические хэши файлов зависимостей. В Go они хранятся в `go.sum` и проверяются при каждой сборке. Это защищает от подмены пакетов в цепочке поставок (supply chain attack).

Кроме того, в Go есть глобальный прокси от Google (`proxy.golang.org`), который хранит копии всех пакетов. Если исходный репозиторий (GitHub) станет недоступен или автор удалит пакет — прокси продолжит отдавать последнюю загруженную версию.

В Java это частично решается через SHA-1 в Maven, но единого глобального прокси нет. В C# ситуация похожая. Go в этом плане более защищён для корпоративной разработки, где надёжность и воспроизводимость критичны.

### Безопасность зависимостей

- `go.sum` — криптографические хэши
- `vendor` — работа без интернета
- `proxy.golang.org` — резервное хранилище
- Отсутствие единой точки отказа (в отличие от npm/PyPI)

# SQL

| Концепция                | Объяснение                                                                 |
| ------------------------ | -------------------------------------------------------------------------- |
| **`SERIAL`**             | Автоинкремент. PostgreSQL сам подставит следующее число.                   |
| **`VARCHAR(n)`**         | Строка с ограничением длины. Используй для имен, email, адресов.           |
| **`TEXT`**               | Строка без ограничения длины. Используй для биографий, описаний, контента. |
| **`DECIMAL(p,s)`**       | Точное число (деньги!). `DECIMAL(10,2)` = до 10 цифр, 2 после запятой.     |
| **`CHECK`**              | Ограничение допустимых значений.                                           |
| **`PRIMARY KEY`**        | Уникальный идентификатор строки.                                           |
| **`FOREIGN KEY`**        | Связь между таблицами.                                                     |
| **`ON DELETE RESTRICT`** | Запрещает удалять родительскую запись, если есть дочерние.                 |
| **`ON DELETE CASCADE`**  | При удалении родителя — удаляются все дочерние записи.                     |

### ПЕРВИЧНЫЕ КЛЮЧИ

| Ключ         | Байт | Макс записей     | Когда использовать          |
| ------------ | ---- | ---------------- | --------------------------- |
| `SERIAL`     | 4    | 2.1 млрд         | Любая обычная таблица       |
| `BIGSERIAL`  | 8    | 9.2 квинтиллиона | Логи, события, трекинг      |
| `INTEGER`    | 4    | 2.1 млрд         | Ручные ID (отделы, типы)    |
| `UUID`       | 16   | Бесконечность    | Микросервисы, безопасные ID |
| `VARCHAR(n)` | n    | —                | Коды (валюты, страны)       |

| Сценарий                                             | Ключ               | Почему                                 |
| ---------------------------------------------------- | ------------------ | -------------------------------------- |
| **Пользователи/Заказы/Товары** (внутренние сущности) | `SERIAL`           | Быстро, дешево, не надо думать.        |
| **Транзакции/Платежи**                               | `UUID`             | Безопасность, глобальная уникальность. |
| **API Ключи/Сессии**                                 | `UUID`             | Нельзя угадать, чтобы взломать.        |
| **Справочники (Статусы/Типы)**                       | `INTEGER` (ручной) | Осмысленные коды (1, 2, 3 или 10, 20). |

1. Если **все равно** на число — `SERIAL`.
2. Если **нужна безопасность/глобальность** — `UUID`.
3. Если **сам проектирую справочник** — `INTEGER` с ручными кодами.

| Сценарий                                              | Какой ключ                                    |
| ----------------------------------------------------- | --------------------------------------------- |
| **Пользователи, заказы, логи, события** (будет МНОГО) | `BIGSERIAL`                                   |
| **Справочники (статусы, типы, страны)** (БУДЕТ МАЛО)  | `INTEGER` (ручной) или `VARCHAR`              |
| **Распределенные системы (микросервисы)**             | `UUID`                                        |
| **Если сомневаешься**                                 | **Всегда выбирай BIGSERIAL** — это безопаснее |
